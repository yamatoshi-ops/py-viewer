# 技術選定検討メモ #2 — Dash 詳細解説（セカンドオピニオン）

## 結論

Dashは**「Pythonだけで、React製のような高機能な解析Webアプリを作れるフレームワーク」**。

---

## 1. Dashとは？

Dashは、グラフライブラリで有名な **Plotly社** が開発した、データ可視化アプリを作成するためのオープンソースフレームワーク。

- **Pythonで完結:** HTML、CSS、JavaScript（React）の知識がなくても、Pythonコードだけで本格的なUIを構築できる
- **Flask + React + Plotly.js:** 内部的には、サーバー側に **Flask**、クライアント側に **React** と **Plotly.js** が使われている。これらをDashが裏側で連携させてくれるため、開発者はデータ処理に集中できる

---

## 2. モータ制御・パワエレ解析におけるメリット

### ◎ SciPyとの強力な連携（FFT・解析）

TypeScript（JS）では貧弱だった解析機能も、Dashならサーバー側で **SciPy** を直接叩ける。

- 窓関数（Hann, Hamming等）を適用した精密なFFT
- THD（全高調波歪）や次数解析の計算
- `signal.spectrogram` を使った時間周波数解析

### ◎ Plotly.jsの資産をそのまま活用

現在のプロトタイプで Plotly.js を使っているため、グラフの設定（`layout` や `data` の辞書形式）の知識がそのまま Python 側で流用できる。

### ◎ 柔軟なUI（Callback機能）

Dashの最大の特徴は **「Callback（コールバック）」**。

- 「スライダーを動かした」→「Python側でFFTを計算し直す」→「グラフが即座に更新される」
- このインタラクティブな動きをPythonの関数として記述できる

---

## 3. Dashアプリの構造

Dashは主に2つの要素で構成される。

1. **Layout (レイアウト):** アプリの見た目（ボタン、グラフ、ドロップダウンの配置）
2. **Callbacks (コールバック):** アプリの動作（入力が変わったときに、どのグラフをどう更新するか）

### コードイメージ

```python
import dash
from dash import dcc, html
from dash.dependencies import Input, Output
import plotly.graph_objs as go
import scipy.fftpack as fft

app = dash.Dash(__name__)

# 1. レイアウト定義
app.layout = html.Div([
    dcc.Graph(id='waveform-graph'),
    dcc.Slider(id='freq-slider', min=1, max=100, value=50)
])

# 2. コールバック（ロジック）
@app.callback(
    Output('waveform-graph', 'figure'),
    [Input('freq-slider', 'value')]
)
```

## 4. 注意点と対策

| 注意点 | 対策 |
|--------|------|
| **サーバーが必要**: TypeScript版（Vite）は静的ファイルとしてブラウザだけで動くが、DashはPythonサーバーが必要 | ローカルで `python app.py` → `localhost:8050` で使う運用なら現在の環境と大きく変わらない |
| **大容量データのオーバーヘッド**: サーバー↔ブラウザ間でデータをやり取りするため、数百万点の生データをそのまま送ると重くなる | 描画前にダウンサンプリング（間引き）を挟むのが定石。非常に重いデータは Panel + Datashader を検討 |

## 6. 重いデータの扱い：データ形式の選定（追記）

モータ制御・パワエレの長時間High-speed loggingでは、CSVはファイルサイズが肥大化し、I/Oだけで数分かかる場合がある。以下のバイナリ形式を検討。

### 推奨データ形式の比較

| 形式 | 圧縮率 | 読込速度 | 特徴 |
|------|--------|---------|------|
| **CSV** | 低（なし） | 遅 | テキストなので視認性は良いが重い |
| **Parquet** | **極めて高い** | **高速** | 解析用途では現在のデファクトスタンダード |
| **HDF5** | 高 | 中〜高 | MATLABの `.mat` と親和性が高く、階層管理が得意 |
| **Feather** | 中 | **最速** | 読み書きのオーバーヘッドが最小。一時キャッシュ用途に最適 |

### 各形式の詳細

**A. Apache Parquet（推奨・採用決定）**
- 圧縮率: CSVと比較して 1/5〜1/10 程度まで削減
- カラムナ形式: 特定チャンネル（例: 電圧だけ）を全体スキャンせずに高速抽出できる
- `pandas.read_parquet()` で爆速読み込み

**B. HDF5 (`.h5`)**
- MATLABの `.mat`（v7.3以降）の実体はHDF5
- 階層構造（フォルダ構造）をファイル内に持てる → 複数の実験データ＋メタデータを1ファイルにまとめられる
- 巨大ファイルでも必要なインデックス範囲だけをメモリにロード可能

**C. Feather**
- 圧縮率より読み書き速度に特化
- ローカルの一時キャッシュに最適

## 7. データ形式変換：MATLAB（.mat）→ Parquet

**採用方針**: Panel_y では **Parquet を標準形式** とする。MATLABデータは前処理として変換する。

### 変換コード例

```python
import pandas as pd
import scipy.io

# 古い形式（v7.2以前）: scipy.io.loadmat を使用
mat_data = scipy.io.loadmat('motor_log.mat')
df = pd.DataFrame({
    'time':    mat_data['time_vector'].flatten(),
    'current': mat_data['iq_data'].flatten()
})
df.to_parquet('motor_log.parquet', compression='snappy')

# 新しい形式（v7.3以降）: h5py または pandas.read_hdf を使用
# df = pd.read_hdf('motor_log.mat', key='dataset_name')
# df.to_parquet('motor_log.parquet', compression='snappy')
```

### 変換のメリット・デメリット

| 比較項目 | HDF5 (.mat) | Parquet |
|---------|-------------|---------|
| **得意なこと** | 複雑な階層構造、行列データの保持 | **列ごとの高速抽出、高い圧縮率** |
| **Python親和性** | ライブラリ経由で読み書き可能 | **Pandasとの相性が最高（標準的）** |
| **Dash/Panelでの読み込み** | やや時間がかかる場合あり | **劇的に速い** |

### 実装上の注意

- **データ型の維持**: Parquetはスキーマ（型情報）を厳密に保持 → `float32` などの精度を維持したまま保存できる
- **メタデータの扱い**: サンプリングレート・実験条件などのメタデータは Parquet のカスタムメタデータ領域、または別ファイル（JSON等）としてセットで管理するのが一般的

## 8. 大容量データの描画対策

Dash でそのまま数百万点をブラウザに送ると Plotly がフリーズする。対策として:

| 手法 | 概要 |
|------|------|
| **LTTB（Largest-Triangle-Three-Buckets）** | 波形の特徴（ピーク）を維持しつつデータ点数を削減するアルゴリズム |
| **Datashader** | Python側で「ピクセル密度」に合わせてデータを集約してから画像をブラウザに送る。Panel との組み合わせで最強 |
| **ダウンサンプリング（間引き）** | 表示倍率に応じて送信点数を制御（Dash Callback 内で実装） |
