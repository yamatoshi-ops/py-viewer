# 技術選定検討メモ #1

## 背景

現行プロトタイプ（TypeScript + Vite + Plotly.js）は TypeScript 学習目的で採用。
FFT解析・コンター図・長時間データ対応を視野に入れた技術スタックを再検討。

---

## 現行スタックの課題認識

| 課題 | TypeScript + Plotly.js の限界 |
|------|-------------------------------|
| FFT | ブラウザJSのFFTライブラリは低機能（窓関数・THD計算が貧弱） |
| 大容量データ | ブラウザのメモリ上限（実質1〜2GB）に縛られる |
| 数値精度 | 浮動小数点精度がSciPyより劣る場面あり |
| 描画速度 | 数十万点を超えると描画が詰まりやすい |

---

## 主要候補の比較

### A. Python + Dash（推奨候補）

```
Python (FastAPI/Flask的な役割) + Plotly Dash (Web UI)
データ処理: NumPy / SciPy / Pandas
```

| 観点 | 評価 |
|------|------|
| FFT品質 | ◎ SciPy.fft → 窓関数・THD・次数解析が揃っている |
| 大容量データ | ◎ NumPy配列でGBクラスも処理可能 |
| 描画速度 | ○ Plotlyと同一API、WebGL対応 |
| コンター図 | ◎ `go.Contour` / `go.Heatmap` がネイティブ対応 |
| 開発コスト | ○ Plotly.jsの知識が流用できる（API体系が近い） |
| 共有・配布 | △ Pythonサーバーが必要 |

**モータ制御用途への強み**: SciPyには `signal.spectrogram` / `signal.periodogram` など制御系で使う解析が全部ある

---

### B. Python + Panel + HoloViews

```
Panel (Bokeh/Plotly/Matplotlib をUIに統合するフレームワーク)
HoloViews: 大容量データの効率的なレンダリング（datashader連携）
```

| 観点 | 評価 |
|------|------|
| FFT品質 | ◎ SciPy利用可能（Dashと同等） |
| 大容量データ | ◎◎ **Datashaderで億点規模も高速レンダリング** |
| 描画速度 | ◎ ピクセル単位の集約描画（長時間データに最強） |
| コンター図 | ◎ HoloViews経由でMatplotlib/Plotly両方使える |
| 開発コスト | △ 学習曲線がDashより高い |
| 共有・配布 | △ サーバー必要（Dashと同じ） |

**特徴**: 長時間波形（数百万点以上）の場合、Datashaderが「画素単位に集約して描画」するため、ブラウザが詰まらない。パワエレ計測で長時間ログを扱うなら最も強い。

---

### C. Python + Streamlit

| 観点 | 評価 |
|------|------|
| FFT品質 | ◎ SciPy利用可能 |
| 大容量データ | ○ メモリ次第だが処理は問題なし |
| 描画速度 | ○ Plotly/Altair/Matplotlib統合 |
| 開発コスト | ◎◎ 最も早く作れる（Pythonだけ書けばいい） |
| 共有・配布 | △ サーバー必要。ただしStreamlit Cloudで無料公開も可 |

**弱点**: インタラクティブUIの自由度が低い（カーソル連動などの細かいUI制御が難しい）

---

### D. Tauri + Python subprocess（デスクトップアプリ）

```
Tauri（Rustシェル）+ Web UI（TypeScript/React）
重い処理はPythonをsubprocessで呼び出し
```

| 観点 | 評価 |
|------|------|
| FFT品質 | ◎ Python/SciPy呼び出し可能 |
| 大容量データ | ◎ ローカル実行なのでメモリ制約が緩い |
| 描画速度 | ◎ ネイティブアプリ、GPU利用可 |
| 開発コスト | ✕ Rust + TypeScript + Python の三重管理 |

現時点ではオーバーエンジニアリング。将来の選択肢として保留。

---

## 選定ポイントの整理

| 選定軸 | 重要度 | 推奨方向 |
|--------|--------|---------|
| FFT・窓関数・THD計算 | 高 | Python必須（SciPy） |
| 長時間データ（数百万点〜） | 高 | Panel + Datashader が最強 |
| コンター図（動作点マップ） | 高 | Python側で処理してPlotlyで描くのが楽 |
| インタラクティブUI（ズーム・カーソル） | 中 | Dash / Panel どちらでも対応可 |
| 開発速度（自分1人で作る） | 中 | Streamlit > Dash > Panel |
| ブラウザなしで使いたい | 低 | Tauri or PyQt方向 |

---

## 結論・推奨

- **長時間データを本格的に扱うなら** → Panel + HoloViews + Datashader
- **まずスピード重視で動くものを作るなら** → Python + Dash

TypeScriptを選んだ理由が「学習目的」であれば、今後の実務用途（パワエレ波形解析）にはPythonスタックの方が圧倒的にフィット。SciPy・NumPyとの連携でFFTもコンターも「書くだけ」で動く。

**段階的移行案**: Phase 1をDashで作り、大容量データで詰まったらPanel + Datashaderに移行。
